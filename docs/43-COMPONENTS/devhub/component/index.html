    DEVHUB - MonE Event Hub

    1. Component Description

    DEVHUB provides a distributed topic and publish/subscribe based Event/Message Streaming Bus. The component makes it possible for the creation of a HUB at the API Level by interested client(s) and notification back to registered callback URLs when event happens. All MonE Components publish their status change events through the MonE DEVHUB provided APIs utilizing the built-in message streaming engine contained in this component.

    Event Hub Management capability allows decoupling of MonE Components, their exposed APIs and also the interested external 3PP applications resulting in high resilience and scalable systems. Each MonE component supports generation of a certain set of events specific to the capability provided by that component. Thus, unless certain events are not disabled, MonE components publish events/messages to a topic, and any other component interested in these messages can subscribe to the topic. Simply, in this model, any message published to a topic is immediately received by all of the subscribers of the topic (unless the receiving agent has applied a message filter pattern).

    In addition to basic publish/subscribe via an Event/Message Streaming Bus, MonE DEVHUB component makes it possible for the creation of a HUB at the API Level compliant to TMFORUM open API(Ref.[7] Part-1)) and notification to registered callbacks. Please refer to Event, Hub and Notification Management Patterns for details.

    Thus, the registration of a listener by an interested client is done by creating a HUB resource unique to the listener by providing a callback URL for notification when the event happens. DEVHUB manages the invocation of callback URLs when the event takes place.

    5. Component Configuration Guide

    This section provides configuration information for the MonE Component.

    Important

    As MonE Components are implemented as containers and deployed as pods onto Kubernetes platform through one of the workload resources managed by Kubernetes (Deployment, DameonSet, Services etc.), the configurative items explained in this section are to be perceived at container level and to be managed via Kubernetes supported resource objects, such as "define an environment variable for a container" or "create config map from a set of config/property files that reside in a folder" etc. To keep the technology neutral component level focus, only vital Kubernetes info is explained here. For details, please refer to Kubernetes Documentation

    5.2. Component Config Files

    All config files reside in ${MonE_ROOT}/${COMPONENT_NAME}/config folder. The config map will be made available to the MonE component via the following Kubernetes facility:

    $kubectl create configmap $COMPONENT_NAME-config --from-file=$MonE_ROOT/$COMPONENT_NAME/configmap/

    The following table depicts the config files used by this component.

    Config File Description

    6.1. Component Kubernetes Deployment/Service Files

    6.2. Management of Deployments/Pods

    How to start Deployment/Pod?

    The following Kubernetes kubectl command may be used to start a new or update and existing Deployment:

    $kubectl apply -f app-deployment.yml

app-deployment/[COMPONENT_NAME] created

    How to check if Deployment/Pod up&running?

    The following Kubernetes kubectl command may be used to check if a Deployment is up&running:

    $kubectl get deployment [COMPONENT_NAME] --namespace=MonE

NAME            READY   UP-TO-DATE   AVAILABLE   AGE
[COMPONENT_NAME]   3/3     3            3           4m6s

    The response indicates that all replicas as configured in deployment are ready and available.

    How to stop/delete Deployment/Pod?

    The following Kubernetes kubectl command may be used to delete a Deployment:

    $kubectl delete deployment [COMPONENT_NAME] --namespace=MonE

app-deployment "[COMPONENT_NAME]" deleted

    6.3. Management of Services (Provided APIs)

    How to start Services?

    The following Kubernetes kubectl command may be used to start the Service:

    $kubectl apply -f app-service.yml

app-service/[COMPONENT_NAME] created

    How to check if Services up&running?

    The following Kubernetes kubectl command may be used to check if a Service is up&running:

    $kubectl get service [COMPONENT_NAME] --namespace=MonE

NAME            TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
[COMPONENT_NAME]   LoadBalancer   10.108.32.34   localhost     80:32108/TCP   79s

    The response indicates that service is up and running.

    How to stop/delete Services?

    The following Kubernetes kubectl command may be used to delete a Service:

    $kubectl delete service [COMPONENT_NAME] --namespace=MonE

app-service "[COMPONENT_NAME]" deleted
