    Functional Features Overview

    This section provides an elaborate product features overview of MonE. To have a 360° understanding of the capability domains that constitute MonE functional landscape, the following diagram illustrates a categorized classification of the main functional domains of Features.


    Figure-1: MonE Functional Feature Domains


    These main functional domains of features are as the following:

    • Open Integration APIs Features: Public Integration APIs capability set provided by relevant MonE components allow integration of MonE Platform to the Enterprise or Service Provider's BSS (Business Support System) Application stack such as; Customer Management Systems, Order Management Systems etc.

    • Monetization Template Model Features: MonE Monetization Template Models enable modeling the financial business associations of the enterprise considering B2C and B2B financial relations and roles of each party involved. Once configured, the Monetization Template Models drive the processing chain utilizing on capabilities provided by Charging Features and Billing Features domains. In this regards, the features in this domain allow setting model driven strategies and rules that determine (including, but not limited to); who pays for what, when, and which type of charges, how charges are calculated from charging metrics and which tariffs to used that derive pricing methods subject to calculations, what B2B parties involved, revenue sharing rules etc.

    • Charging Features: MonE Charging features cover the capabilities provided by the MonE Charging subsystem, such as: supported charging API types and operational modes, charge event filtering/collation/enrichment/aggregation capabilities, charge amount calculation capabilities (flat, tiered, elastic etc.) and supported metrics (QoS, channel, time, zone), basic discounting capabilities, balance management and complex discounting capabilities (free units, cross discounts etc.)

    • Billing Features: MonE Billing features cover the capabilities provided by the MonE Billing subsystem, such as: supported billing operational modes, classification/aggregation capabilities per different financial perspectives (bill line items, taxation, GL posting etc.), bill time applied discounts capabilities

    • Platform Common Functional Features: MonE Billing features relate to common platform functional features, such as; Event Management, Operational Intelligence and Analytical capabilities etc.

    The following sections provide a list of features for each relevant functional domain.

    1. Open Integration APIs Features

    MonE provides public Open Integration APIs that provides decoupling, easy and cost-efficient integration of MonE Platform with the other apps that reside in the Enterprise/Service Provider BSS ecosystem. Open Integration APIs enable integration in the following areas. Please refer to Public APIs Specification for details.

    • Charging APIs
    • Billing Account Management APIs
    • Order Management APIs
    • Financial Transaction Management APIs
    • Balance Management APIs
    • Bill Management APIs
    • Bulk Request Management APIs
    • Financial/Accounting Data Exchange APIs
    • Entity Model Exchange APIs
    • Event Notification APIs

    2. Monetization Template Model Features

    Definition

    Monetization refers to standards and processes defined/adopted for deriving revenue from the value you offer to your customers as wells as the settlement of incoming/outgoing revenue for the involved B2B partners.

    MonE Monetization Template Models refer to a set of configurative conditional contexts that allow the next generation platform business driven Enterprise Organizations or Service Providers to model revenue generation and settlement strategies of different product offerings for B2C and B2B amid the complex ecosystem of service providers and other involved business partners. At the very high level, these MonE Monetization Template Models (MonE MTM) drive the event input, charging, discounting, billing and revenue settlement processing pertaining to specific product offerings.

    The following diagram illustrates the constituents of MonE MTMs. At the very high-level, MonE MTMs are comprised of MonE Mediation MTM, Charging MTM and Billing MTM, in which Charging MTM further consists of PreCharging MTM, B2C Charging MTM and B2B Charging MTM.

    It shall be noted that meanwhile MonE MTMs drive the main information flows, dictating collation/cloning of input events, determination of involved parties and charges to be generated, the additional components within MonE Charging and Billing takes care of the fine-grained processing of these items considering B2C and/or B2B accounts associated tariffs, assigned products, discounts etc..

    Please refer to next sections regarding such features specific to MonE Charging and Billing.

    Figure-2: MonE MTM-Monetization Template Model Features

    2.1. MonE Mediation MTM

    Mediation Templates allow configurable mapping, collation and ingestion of chargeable or billable (pre-charged) events into MonE. A different Mediation MTM can ve configured for each different tuple of (EventInputChannel, FileNamingPattern).

    Once configured the following features are provided via Mediation MTMs that allow generic ingestion of any event into MonE whether for charging and/or billing purposes:

    • Event Qualifiers: This generic mapping allows providing configurable event processing hints to MonE, such as setting; Request Type (PriceRequest, TariffRequest), Pricing Type to be applied (onetime, recurrent, usage) etc.
    • DateTime Quantities: This generic mapping allows providing configurable date time events mapping with role specification of each entity to be further considered by MonE Charging/Billing
    • Involved Charging Metric Data and UoMs (Units of Measurement): This generic mapping allows providing configurable metric info mapping (duration, total/uplink/downlink data volume, number of quantities etc.) with role specification of each entity to be further considered by MonE Charging/Billing
    • Involved ResourceAndAddress Identifiers: This generic mapping allows providing configurable address and resource identifier mapping (served_party, other_party, serving element_address, location_address etc.) with role specification of each entity to be further considered by MonE Charging/Billing
    • Involved Service/Prod Identifiers: This generic mapping allows providing configurable service/product identifier mapping with role specification of each entity to be further considered by MonE Charging/Billing

    2.2. MonE PreCharging MTM

    MonE PreCharging MTMs determine the pre-charging MTM templates to be applied for the specific event type. It shall be noted that only after resolution of the pre-charging model driven criteria, the actual B2C and B2B charging can continue and charging templates can continue to be applied.

    Once the PreCharging MTM RuleSet is executed and the PreCharging MTM to be applied is determined for the relevant input event/stream, the PreCharging MTM provides features that allows configurable determination of the following:

    • Normalization: Involved ResourceAndAddress Identifiers to be normalized and the normalization strategy to be applied (reference address etc.)
    • Product Identification: Involved Product roles and identification keys to be applied for each, additional qualifiers, such as a flag indicating whether identification of the item is mandatory for charging to continue etc.
    • Involved B2C Party Accounts: Involved B2C Account roles and identification keys to be applied for each, additional qualifiers, such as a flag indicating whether identification of the item is mandatory for charging to continue etc.
    • Involved B2B Party Accounts: Involved B2B Account roles and identification keys to be applied for each, additional qualifiers, such as a flag indicating whether identification of the item is mandatory for charging to continue etc.

    2.3. MonE B2C Charging MTM

    MonE B2C Charging MTMs determine the charging MTM templates to be applied for the specific event type. It shall be noted that it is the B2C Charging MTMs that decide how many charges to be created for which of the involved B2C party accounts, using what/whose tariff, applying what charging method etc.

    Once the B2C Charging MTM RuleSet is executed and the Charging MTM to be applied is determined, the B2C Charging MTM provides features that allows configurable determination of the following:

    • Determine number of B2C Charge Items (CI) to be created
    • For EACH B2C CI Determine
      • Set chargedAccount
      • Set payerAccount
      • Set Price Type (OT, REC, USG, etc.)
      • Resolve input quantity/metrics
      • Calculate Charge as configured per Monetization Template charging method: Tariff defined in the system, tariff assigned to an involved party account, or tariff submitted as part of order.
      • For tariff based calculations apply the concluded charge method (drives calculation method: flat, percentage, tiered etc.)

    2.4. MonE B2B Charging MTM

    MonE B2B Charging MTMs determine the charging MTM templates to be applied for the specific event type. It shall be noted that it is the B2B Charging MTMs that decide how many charges to be created for which of the B2B involved party accounts, using what/whose tariff, applying what charging method, whether bill or a settlement note etc.

    Once the B2B Charging MTM RuleSet is executed and the Charging MTM to be applied is determined, the B2B Charging MTM provides features that allows configurable determination of the following:

    • Determine number of B2B Charge Items (CI) to be created
    • For EACH B2C CI Determine
      • Set chargedAccount
      • Set payerAccount
      • Set Price Type (OT, REC, USG, etc.)
      • Resolve input quantity/metrics
      • Calculate Charge as configured per Monetization Template charging method: Tariff defined in the system, tariff assigned to an involved party account, or tariff submitted as part of order. For B2B input may be the charge applied to a B2C account.
      • For tariff based calculations apply the concluded charge method (drives calculation method: flat, percentage, tiered etc.)

    2.5. MonE Billing MTM

    MonE Billing MTMs determine the billing MTM templates to be applied. It shall be noted that it is the Billing MTMs that decide the mechanical aggregation/classification rules that relate to bill line items aggregation, tax items aggregation, GL code aggregation etc.

    Once the Billing MTM RuleSet is executed and the Billing MTM to be applied is determined, the Billing MTM provides features that allows configurable determination of the following:

    • For EACH (Payer Acct, Charge Item)
      • Determine Billing MTM to be Applied
      • Determine/override billingAccount as per Split billing rules assigned to account. If no rules, billedAccount=chargedAccount
      • Rules for Bill Line Item Aggregation
      • Rules for Accounting/GL Item Aggregation
      • Rules for Tax/Item Aggregation
      • Rules for External Cost Center Aggregation

    2.6. Other Enhanced Features

    In addition to many flexible and unique features supported by MonE aforementioned in above sections, the following are (including, but not limited to) some other critical enhanced features that allows a better perception of the value MonE brings into the enterprise business:

    • Elastic/Dynamic Pricing: A pricing where MonE Charging Tariff decision tree dictates a varying rate calculation method as a function of external load demand separate to predetermined factors like location/time/date/usage
    ChargeAmount = f(externalSupply, externalDemand)
    • Blended Rate: A billing time applied function that allows overriding the aggregated amount for a specific bill line item. The amount is determined as a function of calculating a weighted average over multiple periods that allows smoothing of outliers.
    InvoiceItemAmount = W*Avg(itemX, numberOfPeriods)

    • External Cost Center Allocation: A billing time applied additional classification and aggregation function that can be sold as a value added product allowing classifying charged items based on an explicitly assigned external cost allocation plan. The result is a customer/partner driven cost allocation report pertaining to internal projects codes of the business partner in addition to legal financial invoice.

    • Best Option Plan: A billing time applied value added function that can be sold as a value added product (supported by charging) that allows assigning additional tariffs to a product and parallel charging of items and let billing choose the best option for a particular bill line item while generating the bill.

    • Credit/Cost Control Quotas and Alerts: Configurable rating and billing time applied cost control and alert mechanisms that prevent bill shocks.

    • Enhanced Commitment Plans: Even though from an ordering and sales perspective, commitments in general indicate duration based commitments with a fixed procedure of penalty calculation if breached, MonE enables managing Commitments with considering the following perspectives:

      • Duration based commitments
      • Revenue based commitments as per revenue generated during a defined period (month, quarter, year, billcycle etc.)
      • Flexible Actions on breach: Apply fixed penalty charge, percentage based penalty charge referring to another charge item, reverse benefits gained via related discount product etc.

    • Enhanced Installment Plans: Order submitted with a related installment plan package can be billed as per the configured pay schedule.

    • Flexible Partner Billing/Settlement Methods: Percentage based, Flat, Transactional Tiered, Aggregated Revenue Tiered, Settlement Immediated, Settlement Upon Payment etc.

    5. Platform Common Functional Features

    This section details common functional features utilized by all other functional blocks.

    5.1. Event, Communication Hub and Notification Management Features

    The following is a list of critical functional features supported by this domain:

    • Management of Topic Resources that allow publishing Event resources on a topic
    • Event Management, publishing of a certain set of Events to Topics specific to the capability provided by each MonE component
    • Management of Communication Hub resources, bound to a Topic resource, that allows registration of listeners for particular events.
    • Notification Management to registered callback Listener URIs, where Events that will be POSTed/notified

    For details, please refer to Event, Hub and Notification Management Patterns within Public API Guidelines.

    5.2. Intelligence and Information Management Features

    The following is a list of critical functional features for this domain:

    • All MonE Components generate Operational Information stream providing data regarding the operational status of the component including, but not limited to: timestamp, component name, thread info, transactional status information
    • All MonE Components generate KPI Statistics Information stream providing data regarding the statistics of the component including, but not limited to: per transaction type total number of success, failures, mean response time, min/max response times
    • Collection and aggregation of operational and statistical information
    • Dashboards for Operational and KPI status monitoring

    For details, please refer to L2 Architecture.
