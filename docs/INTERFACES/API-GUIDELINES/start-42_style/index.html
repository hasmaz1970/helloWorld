    URL Path Structure and API Style

    This section details the naming and JSON style guidelines supported by MonE APIs.

    1. API URL Path Structure and Versioning

    The following pattern constitutes the MonE service endpoint URL Path structure including versioning, in which [optional] imply the optional parts.

    {protocol}://{serverRoot}[:port]/{apiRoot}/{VERSION}/{resourceName}/[{resourceID}]

    Example

    The following is an example WITHOUT the [optional] parts.

    https://bss.operator.com/MonE/customerManagement/v4/customer where;

    • customer denotes Resource Name


    The following is an example INCLUDING the [optional] parts.

    https://bss.operator.com:8080/MonE/customerManagement/v4/customer/42 where;

    • customer denotes Resource Name
    • 42 denotes the resource Identifier


    API Versioning adheres to the following principles:

    • MonE APIs state version with “v” following the API Name, e.g.: APIName/v1/resource.
    • Schema associated with has its version number aligned with that of the API.
    • The version number has major, minor and revision numbers. E.g. v1.0.0
    • Only the major version number is held in the URI. E.g customerManagement/v1/customer.
    • Any change that causes backward incompatibility results in an increment of the version number:
    • Major changes are such as an existing sub part structure change or element type changes resulting in the client logic to break.
    • Minor backward compatible changes are such as; new optional elements added, cardinality of attributes from mandatory to optional or from lower to greater etc. do NOT impact backward compatibility and does not result in major number increment.

    2. API Resource Archetypes

    MonE REST APIs resource archetypes align each resource to just one of these:

    • Collection Resource – set or collection of resources. A resource with no identifier represents the Collection Resource, such as https://bss.operator.com/MonE/customerManagement/v4/customer
    • Managed Resource – an object instance (that may represent a persisted database record etc.). A resource with identifier represents Managed Resource. Its representation includes: fields with values and links to related resources. Can have child resources, of different resource types. Client can create, query, update and delete resources. Example: https://bss.operator.com/MonE/customerManagement/v4/customer/42
    • Task Resource – resources that are executable functions – with associated input and output parameters. Necessary where the required action cannot be mapped to standard CRUD methods. Tasks play the role of Controllers e.g. ProductOfferingQualification, ServiceQualification. Example: https://bss.operator.com/MonE/productOfferingQualification/v4/productOfferingQualification

    3. Resource Name Guidelines

    Resource names must conform to the following guidelines:

    • Resource names should be meaningful names with defined semantics, derived from TMF Open APIs and Information Framework Resource name MUST NOT expose technical or implementation details. For example, resourceSpecification, NOT resourceSpec
    • Resource names must be camel case, ascii strings.
    • Each Collection Resource or Managed Resource name must be a NOUN, except for Tasks which must be a verb.
    • Collection Resource names MUST NOT have a specific notation, such as "Collection" postfix or plural form. For example, API/ticket represents the Ticket Collection and GET API/ticket retrieves all the items from the Ticket Collection (i.e. all the resources of type Ticket contained within the application). So forms like API/ticketCollection or API/tickets MUST NOT be used.

    4. JSON Style and Resource Fields (Property)

    4.1. Definitions

    The readers familiar with JSON syntax can skip this section.

    JSON syntax is derived from JavaScript object notation syntax:

    • Data is in name/value pairs
    • Data is separated by commas
    • Curly braces hold objects
    • Square brackets hold arrays

    For the purposes of this style guide, the definitions of the following terms are important:

    • resource field (aka property) - a name/value pair inside a JSON object.
    • field name (aka property name) - the name (or key) portion of the property.
    • field value (aka property value) - the value portion of the property.

    Example

    The following is a simple JSON object specifying values of field/property names: name and surname 

    {
  "name": "John",
  "lastName": "Doe"
}

    Example

    The following is a more complex conceptualJSON object specifying a conceptual resourceX that is related to two items 

    {
  "id": "123",
  "href": "https://bss.operator.com:8080/MonE/xyzManagement/v4/resourceX/123",
  "description": "resourceX",
  "isActive": true,
  "lastUpdate": "2020-07-01T06:11:00.281Z",
  "lifecycleStatus": "retired",
  "name": "resourceX_123",
  "version": "v1",
  "attachment": [
    {
      "id": "54",
      "attachmentType": "file",
      "mimeType": "application/json",
      "name": "file54.json",
      "url": "ftp://ftp.myCatalog.com/itemCatalog/54"
    },
    {
      "id": "55",
      "attachmentType": "file",
      "mimeType": "application/json",
      "name": "file55.json",
      "url": "ftp://ftp.myCatalog.com/itemCatalog/55"
    }
  ]
}

    4.2. Resource Field (Property) Naming Guidelines

    Property names must conform to the following guidelines:

    • Property names should be meaningful names with defined semantics.
    • Property names must be camel case, ascii strings.
    • The first character can be a letter, an (at sign) "@", (an underscore) "_" or (a dollar sign) "$".
    • Subsequent characters can be a letter, a digit, an underscore, or a dollar sign.

    4.3. Resource Field (Property) Value Guidelines

    Field values must contain one of:

    • a string
    • a number
    • a JSON object
    • an array

    • a boolean

    • Property values conform the JSON spec, which details exactly what type of data is allowed in a property value. This includes Unicode booleans, numbers, strings, objects, arrays, and null.

    • JavaScript expressions are not allowed.
    • MonE APIs choose the data type most appropriate for a particular property (numbers to represent numbers, etc.). Javascript's number type encompasses all floating-point numbers, which is a broad designation. Numbers refer to JavaScript's number type, while integer will refer to integers.

    Good example ( // are for comments only and NOT part of the JSON): 

    {
  "canCowsFly": null,   // null
  "isThisOK": false,    // boolean
  "yourID": 42,         // number-integer
  "actualValue": 79.82, // number-float  
  "name": "John",       // string
  "moreData": {         // object
      "dataVal1": 12,
      "dataVal2": 22
  },
  "things": [
      "thing1", "thing2"
  ]           
}

    Bad example: 

    {
  "aVariableName": aVariableName,         // Bad - JavaScript identifier
  "functionFoo": function() { return 1; }    // Bad - JavaScript expressions not allowed
}

    4.5. Date and DateTime Property Values

    Dates are formatted as recommended by RFC 3339 and conformant to ISO 8601 as the following:

    • YYYY-MM-DDThh:mm:ss.SSSTZD
    • YYYY = four-digit year
    • MM = two-digit month (01=January, etc.)
    • DD = two-digit day of month (01 through 31)
    • T = optional delimiter
    • hh = two digits of hour (00 through 23) (am/pm NOT allowed)
    • mm = two digits of minute (00 through 59)
    • ss = two digits of second (00 through 59)
    • SSS = up to 3 digits representing a decimal fraction of a second

    • TZD = time zone designator (Z or +hh:mm or -hh:mm)

    • All the following examples will be supported. Unless otherwise specified the default will be “dateTime”.

    • If the southbound returns the time in epoch time format and the value returned is 0 then the relevant fields associated with the southbound attribute should be shown as blank in MonE Northbound API.
    {
  “date”:       “2014-02-20”
  “dateTime”:           “2014-02-20T16:30:00”,
  “dateTimeGMT”:    “2014-02-20T16:30:00GMT+5”,
  “dateTimeStamp”:  “2014-02-20T16:30:00.135GMT+5”,
  “dateTimeStampGMT”:   “2014-02-20T16:30:00.135GMT+5”
}

    4.6. Time Duration Property Values

    By default, unless otherwise specified in API documentation, time duration values are in milliseconds. 

    “executionTime”: “2321”     // 2 seconds and 321 milliseconds

    4.7. Latitude/Longitude Property Values

    Latitudes/Longitudes are formatted as recommended by ISO 6709. Furthermore, they favor the following formats

    Latitude Latitude is a number preceded by a sign character. A plus sign (+) denotes northern hemisphere or the equator, and a minus sign (-) denotes southern hemisphere.

    Longitude Longitude is a number preceded by a sign character. A plus sign (+) denotes east longitude or the prime meridian, and a minus sign (-) denotes west longitude or 180° meridian (opposite of the prime meridian).

    For specifying geolocation information, the following approach is used in MonE APIs: 

    {
  // The latitude/longitude location of the statue of liberty.
  "statueOfLiberty": {
          “location”: {
                "lattitude": "+40.6894",
                "longitude": "-074.0447",
                "altitude": "5",         // optional
                "uncertainty": "100"     // optional
          }
   }
}

    4.8. Empty/Null Property Values

    If a property is optional or has an empty or null value, by default it will be dropped from the JSON, unless there's a strong semantic reason for its existence.

    {
  "volume": 10,

  // Even though the "balance" property's value is zero, it should be left in,
  // since "0" signifies "even balance" (the value could be "-1" for left
  // balance and "+1" for right balance.
  "balance": 0,

  // The "currentlyPlaying" property will by default be left out since in response since it is null.
  // "currentlyPlaying": null
}
