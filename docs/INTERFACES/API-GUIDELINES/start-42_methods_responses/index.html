    Methods, Response Codes and Error Representation

    This section describes the MonE REST API regarding supported operations and response codes.

    1. API Methods (http-Methods)

    MonE API uses the http-methods as depicted in the following table and detailed in [RFC2616]:

    http-METHOD Purpose Description
    GET Query Entities GET must be used to retrieve a representation of a resource whether be collection or a managed entity resource
    HEAD GET meta information in HTTP headers only The HEAD method is identical to GET except that the response does NOT contain a message-body.
    This METHOD is optional and may NOT be supported by some MonE APIs, in which case the very similar result can be achieved by using "fields=none" directive as part of GET requests (See below sections:TODO)
    POST Create Entity (for a Managed Resource) POST must be used to create a new resource
    POST Execute an Action on an Entity (for a Task Resource) PUT must be used to execute/trigger an action on a Task Resource identified by its resource URI.
    PUT Complete Update of an Entity with all its sub parts PUT must be used to completely update a resource identified by its resource URI. Note that any subpart of the resource not provided as part of PUT, means nullifying its value and will be LOST after the update.
    PATCH Partial Update of an Entity for the provided sub parts PATCH must be used to partially update a resource identified by its resource URI. Note that ONLY provided subpart of the resource will be modified, the others remaining not changed after a PATCH.
    DELETE Remove an Entity DELETE must be used to delete a resource identified by its resource URI. Note that the back-end API functionality may be marking the entity as DELETED instead of physical removal, still resulting in the same but operationally a more secure effect.

    Warning

    GET and POST must not be used to tunnel other request methods.

    2. HTTP Response/Status Codes Overview

    REST APIs use the Status-Line part of an HTTP response message to inform clients of their request’s overarching result. [RFC2616] defines the Status-Line syntax as shown below. Please note that the following is the same whether application layer protocol HTTP/1.1 is used without protocol security or encrypted using TLS.

    HTTP Response Status-Line as per [RFC2616] is as the following:

    HTTP-Version SP Status-Code SP Reason-Phrase CRLF

    where:

    • HTTP-Version: is HTTP protocol version
    • SP: space
    • Status-Code: is HTTP Status Code
    • Reason-Phrase: is the phrase associated with Status-Code, intended to give a short textual description of the Status-Code
    • CRLF: is carriage return followed by line feed

    Some example(s) are: 

    * HTTP/1.1 200 Success
* HTTP/1.1 206 Partial Response
* HTTP/1.1 400 Bad Request

    The following table depicts a high-level list of responses codes aligned with standard HTTP conventions, also followed by MonE APIs:

    • 2xx indicate Success
    • 3xx redirection
    • 4xx indicate Client Error(s)
    • 5xx indicate Server Error(s)

    3. MonE API Response/Status Codes

    Following industry best practice patterns, MonE API response codes whether be success or error, aligns with TMFORUM Open API Guidelines (Ref. [1], Part-1).

    Please note that in order to keep the original semantics of [RFC2616] specification serving as the basis for http-REST style API implementations, the "server" term is used in the response code descriptions. With regards to MonE, this means the MonE component providing the relevant API packaged as a container and deployed on container orchestration platform, such as Kubernetes.

    The following sections depict status code(s) supported by MonE APIs.

    3.1. Success Status Codes:2xx

    Status Code Reason-Phrase Description
    200 OK OK - Used to indicate nonspecific success. Must not be used to communicate errors in the response.
    201 Created Created - Used to indicate successful resource creation. Return message SHOULD contain a resource representation and a Location header with the created resource’s URI.
    202 Accepted Accepted - Used to indicate successful start of an asynchronous action, indicating that the submitted request has been successfully accepted by the server.
    204 No Content in Response / Deleted No Content / Deleted - Used when the response body is intentionally empty indicating that the MonE API provider has fulfilled the request but does not need to return an entity-body, and returned updated meta information only (response with HTTP-headers). In response to DELETE, this status code is used unless otherwise else specified.
    206 Partial Content Partial Content – Used in response to GET indicating partial resource returned in response (with pagination).

    3.2. Redirection Status Codes:3xx

    Status Code Reason-Phrase Description
    301 Moved Permanently Moved Permanently - Used to relocate resources. The requested resource has been assigned a new permanent URI and any future references to this resource SHOULD use one of the returned URIs.
    303 See Other See Other - In HTTP standards, this status code and method exist primarily to allow the output of a POST-activated script to redirect the user agent to a selected resource. In MonE API(s), this may be used to refer the client to a different URI – can be used with a Location header containing the URI of a resource that shows the outcome of an asynchronous task.
    304 (not supported, considered for future) Not Modified Not Modified - Used to preserve bandwidth in the case of conditional gets, which requires additional HTTP Request Header directives: If-Modified-Since, If-Unmodified-Since, If-Match, If-None-Match, or If-Range. Simply, if the client has performed a conditional GET request and access is allowed, but the document has not been modified, the server respond with this status code, and the response without message-body.
    307 Temporary Redirect Temporary Redirect - Used to tell clients that the requested URI resides temporarily under a different URI and the client should resubmit the request to this temporary URI.

    3.3. Client Error Status Codes:4xx

    Status Code Reason-Phrase Description
    400 Bad Request Bad Request - (Syntax Error in Request Body) The request could not be understood by the server due to malformed syntax. The client SHOULD NOT repeat the request without modifications.
    401 Unauthorized Unauthorized - Used when there is a problem with the client’s credentials
    403 Forbidden Forbidden - The server understood the request, but is refusing to fulfill it. Authorization will not help and the request. For example, a client may be authorized to interact with some, but not all of a REST API’s resources. If the client attempts a resource interaction that is outside of its permitted scope, the REST API responds respond with 403.
    404 Not Found Not Found - The server has not found anything matching the Request-URI. No indication is given of whether the condition is temporary or permanent. This error code is used when a client’s provided URI cannot be mapped to a resource URI.
    405 Method Not Allowed Method Not Allowed - The method specified in the Request-Line is not allowed for the resource identified by the Request-URI
    406 Not Acceptable Not Acceptable - The resource identified by the request is only capable of generating response entities which have content characteristics not acceptable according to the accept headers sent in the request.
    409 Conflict Conflict - The request could not be completed due to a conflict with the current state of the resource. This code is only allowed in situations where it is expected that the user might be able to resolve the conflict and resubmit the request. Conflicts are most likely to occur in response to a updated (PUT/PATCH) requests. For example, if versioning were being used and the entity being PUT included changes to a resource which conflict with those made by an earlier (third-party) request, the server might use the 409 response to indicate that it can't complete the request.
    410 Gone Gone - The requested resource is no longer available at the server and no forwarding address is known. This condition is expected to be considered permanent. Clients (if keeping remote links) may delete references to the Request-URI. PLease note that if server does not know, or cannot determine, whether or not the condition is permanent, then simply returns status code 404 (Not Found) instead.
    411 Length Required The server refuses to accept the request without a defined Content-Length HTTP Header in request.
    412 Precondition Failed Not supported, planned for future releases.
    Precondition Failed - The precondition given in one or more of the request-header fields evaluated to false when it was tested on the server.
    413 Request Entity Too Large Request Entity Too Large - The server is refusing to process a request because the request entity is larger than the server is willing or able to process.
    414 Request-URI Too Long Request-URI Too Long - The server is refusing to service the request because the Request-URI is longer than the server is willing to interpret
    415 Unsupported Media Type Unsupported Media Type - The request entity has a media type which the server or resource does not support. For example, the client uploads an image as image/svg+xml, but the server requires that images use a different format
    422 Unprocessable Entity Unprocessable Entity - (Semantic Error in Request Body) The request body contains well-formed (i.e., syntactically correct), but semantically erroneous context (which means neither 415-Unsupported Media Type nor 400-Bad Request applies to this error).
    429 Too Many Requests Too Many Requests – SHOULD be used to indicate that the client has sent to many requests in a given amount of time (“rate limiting”).
    This response code is indirectly supported as a side car capability to MonE component APIs via container platform provided load balancing/throttling capabilities resource, such as nginx/ingress.

    3.3. Server Error Status Codes:5xx

    Status Code Reason-Phrase Description
    500 Internal Server Error Internal Server Error – a generic error message, given when an unexpected condition was encountered preventing the server from providing fulfilling the request.
    501 Not Implemented Not Implemented - The server does not support the functionality required to fulfill the request. This is the appropriate response when the server does not recognize the request method and is not capable of supporting it for any resource.Usually this implies future availability (e.g., a new feature of a web-service API)
    502 Bad Gateway Bad Gateway - The server, may be acting as a gateway, proxy, or broker being dependent on other server(s) APIs (aka required APIs) to fulfill the request. This response code indicates that the server received an invalid response from the other upstream server(s) while attempting fulfill the request. Please also see 504
    503 Service Unavailable Service Unavailable - The server is currently unavailable (because it is overloaded or down for maintenance). Generally, this is a temporary state.
    504 Gateway Timeout Service Unavailable - The server, while acting as a gateway or proxy, did not receive a timely response from the upstream server(s) while attempting to fulfill the request.

    4. Error Representation

    MonE APIs adhere to TMFORUM API Guidelines recommendations for the representation of detailed application specific error objects, but also extends the TMFORUM recommended Error Representation with Industry best practices (Google APIs) to include additional informative information in error responses.

    The following table depicts the Error Representation in body of the response returned by MonE APIs.

    Property Description Mandatory (M) /
    Optional (O)
    code Application related code (as defined in the API or from a common list) M
    reason Text that explains the reason for error. This can be shown to a client user. M
    message Text that provide more details and corrective actions related to the error. This can be shown to a client user. O
    status http error code extension like 400-2 O
    referenceError url pointing to documentation describing the error O
    @baseType When sub-classing, this defines the super-class. O
    @schemaLocation A URL to a JSON-Schema file that defines additional attributes and relationships. O
    @type When sub-classing, this defines the sub-class entity name. O
    domain Domain of the error, such as:
    • Integration APIs
    • Charging
    • Billing
    • Database
    • In-Memory Cache
    • Message Bus
    		 O
    source Source component name that raised the error, such as:
    • C-FMF
    • API-ORDER-MGMT
    • MonE_DB
    		 O
    locationType locationType of the error, such as:
    • request.header, request.parameter, request.body
    • database-sql
    • etc.
    		 O
    location If available more details on the location of the error. O


    The following are some examples:

    Example-1

    {
  "code": 400,
  "reason": "Bad Request",
  "message": "Invalid sort directive parameters.",
  "domain": "Integration APIs",
  "component": "API-ACCOUNT-MGMT", 
  "locationType": "parameter",
  "location": "sort"
 }
}

    Example-2

    {
  "code": 401,
  "reason": "Unauthorized",
  "message": "Invalid credentials.",
  "domain": "Integration APIs",
  "component": "API-ACCOUNT-MGMT", 
  "locationType": "parameter",
  "location": "apiKey"
 }
}
