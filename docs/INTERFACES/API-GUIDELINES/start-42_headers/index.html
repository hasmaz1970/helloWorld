    HTTP Headers

    This section provides information with regards to headers as supported by MonE API. In web-based solutions, HTTP headers allow the client and the server to pass additional information with the request or the response. An HTTP header consists of its case-insensitive name followed by a colon ':', then by its value (without line breaks). Leading white space before the value is ignored. Thus, headers can be grouped according to their contexts:

    • General HTTP Headers: Headers applying to both requests and responses but with no relation to the data eventually transmitted in the body.
    • Request HTTP Headers: Headers containing more information about the resource to be fetched or about the client itself. Request HTTP headers are submitted by the client in requests. The superset of such HTTP headers must comply to as specified in [RFC2730], [RFC2731], [RFC2732],[RFC2734],[RFC2735].
    • Response HTTP Header: Headers with additional information about the response, like its location or about the server itself (name and version etc.). Response HTTP Headers are provided by the server in responses. The superset of such HTTP headers must comply to as specified in [RFC2730], [RFC2731], [RFC2732],[RFC2734],[RFC2735].
    • Entity HTTP Headers: Headers containing more information about the body of the entity, like its content length.

    The following sub sections provide details for each HTTP Header specific to MonE API.

    Attention

    As per RFC2616, header information is case-insensitive.

    1. General headers

    1.1. Cache-Control

    The following response Cache-Control Header directives are supported by MonE APIs as per [RFC7234] with 200 ("OK") responses to http GET requests.

    • must-revalidate: This directive indicates that once it has become stale, a cache MUST NOT use the response to satisfy subsequent requests without successful validation on the origin server (MonE Component API). Default: must-revalidate is set
    • max-age: This directive indicates that the response is to be considered stale after its age is greater than the specified number of seconds. Default: 60
    • no-cache: This directive indicates that the response MUST NOT be used to satisfy a subsequent request without successful validation on the origin server (MonE Component API). This allows an origin server to prevent a cache from using it to satisfy a request without contacting it, even by caches that have been configured to send stale responses. Default: non-cache is not set

    Example

    HTTP/1.1 200 OK
Content-Type: application/json 
Cache-control: max-age=60, must-revalidate

    1.2. CorrelationContext

    correlationContext is an optional http-Header directive supported by all MonE Public APIs that contains a collection of related properties. Once provided by external API client, correlationContext context is carried into MonE operational LOG streams and also echoed back in response http-Header for all 2xx, 3xx and 4xx responses. Even though not mandatory, providing all the properties of correlationContext is vital; enables correlation of MonE operations to business transactions initiated by external applications/process management platforms; enables enriched analytical capabilities, such as: time profiling, root cause analysis etc.
    correlationContext http-Header object contains the following properties.

    site: All requests to MonE APIs may contain site that may be provided by external API client. site identifier denotes a specific site specific info.

    Example:

    • "West-Coast"
    • "Texas"

    domain: All requests to MonE APIs may contain domain that may be provided by external API client. domain identifier denotes a specific application domain info.

    Example:

    • "Customer Management"
    • "Order Management"

    channel: All requests to MonE APIs may contain channel that may be provided by external API client. channel identifier denotes a specific channel info. The channel may be used to indicate a specific business sub market that may be used for correlation purposes.

    Example:

    • "Walmart"
    • "Lakers"

    source: All requests to MonE APIs shall contain source provided by external API client. source identifier denotes a specific Enterprise Application that issues a series of MonE Public API calls.

    Examples:

    • For an Enterprise Application "OrderManagement-AppA", deployed on West Coast, it may be orchestrated (via an API Manager) to issue MonE API calls with source=XYZ-Billing
    • For a Business Management Platform "BPM-A-CustomerCare", it may be orchestrated (via an API Manager) to issue MonE API calls with source=BPM-A-CustomerCare

    source is a free format string, such as: OrderManagement-AppA, BPM-A-CustomerCare. If NOT provided, it will be populated as NULL into MonE LOG streams limiting end2end operational analytics capabilities.

    sessionId: All requests to MonE APIs shall contain sessionId provided by external API client. sessionId identifier denotes a Business Session of an Enterprise User/Application Account having logged in/accessed an Enterprise Application/Domain that results a series of MonE Public API calls until logout/end of access.

    Example:

    • For an Enterprise Application "OrderManagement-AppA", deployed on West Coast, and for Enterprise User "John Doe" having logged in on 2020-10-10 12:30:00 PST until logout on 2020-10-10 17:30:00, it may be orchestrated (via an API Manager) to issue MonE API calls with the same unique sessionId=df592340-ff9a-11ea-adc1-0242ac120002 (sourceId=OrderManagement-AppA)

    SessionId can be a free format string; typically compliant to [RFC4122], sessionId is recommended to conform to UUID universally unique identifier as a 128-bit number, such as:
    df592340-ff9a-11ea-adc1-0242ac120002. If NOT provided, it will be populated as NULL into MonE LOG streams limiting end2end operational analytics capabilities.

    interactionId: All requests to MonE APIs shall contain interactionId provided by external API client. interactionId identifier denotes a Business Interaction within a Business Session that results a series of MonE Public API calls all related to the same customer or account.

    Example:

    • For an Enterprise Application "OrderManagement-AppA", deployed on West Coast, and for Enterprise User "John Doe" having logged in on 2020-10-10 12:30:00 PST until logout on 2020-10-10 17:30:00, and having served for billingAccount-A, billingAccount-B and again billingAccount-A respectively and each resulting in 3 MonE API calls, it may be orchestrated (via an API Manager) to issue MonE API calls with (sourceId=OrderManagement-AppA, sessionId=df592340-ff9a-11ea-adc1-0242ac120002):
      • interactionId=1111a054-ff9b-11ea-adc1-0242ac120002 for the first set of three MonE API calls related to billingAccount-A
      • interactionId=2222a054-ff9b-11ea-adc1-0242ac120002 for the second set of three MonE API calls related to billingAccount-B
      • interactionId=3333a054-ff9b-11ea-adc1-0242ac120002 for the third set of three MonE API calls related to billingAccount-A
      • etc.

    InteractionId can be a free format string; typically compliant to [RFC4122], interactionId is recommended to conform to UUID universally unique identifier as a 128-bit number, such as:
    3333a054-ff9b-11ea-adc1-0242ac120002. If NOT provided, it will be populated as NULL into MonE LOG streams limiting end2end operational analytics capabilities.

    transactionId: All requests to MonE APIs shall contain transactionId provided by external API client. transactionId identifier denotes an operational unique Transaction Number within a Business Interaction for a MonE Public API call.

    TransactionId can be a free format string; typically compliant to [RFC4122], transactionId is recommended to conform to UUID universally unique identifier as a 128-bit number, such as:
    788ca97e-56dd-44ad-ae0b-6edbb2759ec6. If NOT provided, a UUID will be generated by MonE and will be echoed back in response http-Header to calling client.

    3. Request headers

    The following are critical HTTP headers to be provided by the MonE API clients as part of requests.

    • Accept: This header informs the informs the server (MonE component that provides the relevant API) about the media type(s) required and supported by the client. MonE API(s) supported media type is application/json and charset is utf-8. If a media type is specified that is not supported by the server then the response returned by MonE API will be using the default, application/json, i.e. as if no media type was specified in header. Default: application/json; charset=utf-8. Please note that charset directive is optional, default encoding being utf-8.
    • Accept-Language: This header informs the server (MonE component that provides the relevant API) about the human language the server is expected to send back. If a language is specified that is not supported by the server then the response returned by MonE API will be using the default, en_GB, i.e. as if no language type was specified in header. Default: en_GB
    • Authorization: MonE supports API Key based authentication used together with HTTPS for public APIs. An API key is a unique identifier token/secret that a client provides as part of http-REST call parameter when making MonE Public API calls. Authorization is only used in cases of OAuth. In this case, The HTTP Authorization request header contains the credentials to authenticate/authorize API user agent with a server, usually, but not necessarily, after the server has responded with a 401 Unauthorized status and the WWW-Authenticate response header. Authorization header value contains TYPE_space_CREDENTIALS where;
      • TYPE: Indicates authentication method type, such as Basic or Bearer
      • space: space
      • CREDENTIALS: Indicates credentials context required for authentication/authorization depending on type. For details regarding MonE API supported security capabilities, please refer to Protocol Security.


    Example

    ...

Accept: application/json; charset=utf-8
Accept-Language: en_GB

...

    4. Response headers

    The following are critical HTTP headers to be provided to to client requests in MonE API responses.

    • WWW-Authenticate: The HTTP WWW-Authenticate response header is sent along 401 Unauthorized response and defines the authentication method that should be used to gain access to a resource. WWW-Authenticate header value contains TYPE_space_REALM where;
      • TYPE: Indicates authentication method type, such as Basic, Bearer or OAuth
      • space: space
      • REALM: In the form of "realm=[realm context]" provides a description of protected area. For details regarding MonE API supported security capabilities, please refer to Protocol Security.
    • Location: Typically provided when served with 201 (Created) in response to resource creation, or when served with 3xx (redirection). It provides created resource URL or redirection URL. This is not a guaranteed supported response header by MonE APIs for resource creation. For resource creation responses, instead the reference URL in response payload shall be used by the client.
    • ETag: This header allows caches to be more efficient, and saves bandwidth, as a web server does not need to send a full response if the content has not changed. On the other side, if the content has changed, ETag(s) are useful to help prevent simultaneous updates of a resource from overwriting each other ("mid-air collisions"). If the resource at a given URL changes, a new Etag value must be generated. ETag(s) are therefore similar to fingerprints and might also be used for tracking purposes by some servers. A comparison of them allows to quickly determine whether two representations of a resource are the same, but they might also be set to persist indefinitely by a tracking server.
    • X-Total-Count: Regarding query of Collection Resource types, the response is an array of entities. For query-based pagination, the HTTP header “X-Total-Count” in a response is returned, with the total number of matching resources so that the client can calculate the next page.
    • Link: This header allows response information to contain a comma separated list of href and rel (relation) information. This header is used in responses, such as for paginated queries. This header allows a way for the API server to tell the API client what HTTP requests the client might want to make in the future. It is like a menu, provided by the server, from which the client is free to choose.

    Example

    206 Partial Content
Content-Type: application/json
X-Total-Count:9
Link: < /geographicAddressManagement/validate/8313/iterator?offset=3&limit=3>; rel="self",
< /geographicAddressManagement/validate/8313/iterator?offset=0&limit=3>; rel="first",
< /geographicAddressManagement/ validate/8313/iterator?offset=6&limit=3>; rel="next", 
< /geographicAddressManagement/ validate/8313/iterator?offset=0&limit=3>; rel="prev", 
</geographicAddressManagement/ validate/8313/iterator?offset=6&limit=3>; rel="last"
...

    5. Entity HTTP headers

    The following are critical HTTP headers that are about the body of the entity.

    • Content-Length: If in response, this header informs the client about the length of response in decimal number of octets//bytes. If in request, this header informs the MonE API provider about length of the message-body in the request message.
    • Content-Type: This header informs the informs the client about the media type of the resource in response. MonE API(s) supported media type is application/json and charset is utf-8.Default: application/json; charset=utf-8. Please note that charset directive is optional, default encoding being utf-8.
    • Content-Language: This header informs the client about the about the human language the server is expected to send back. Default: en_GB

    Example

    HTTP/1.1 200 OK 
Content-Type: application/json; charset=utf-8
Content-Language: en_GB 
Content-Length: 1024
ETag: “customer_24_a11”
