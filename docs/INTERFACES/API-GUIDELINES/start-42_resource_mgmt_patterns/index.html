    Resource Management Patterns

    1. Create Resource Patterns

    1.1. Creating a single resource

    http-POST method is used for creation of a single resource. The following design guidelines apply for this pattern:

    • Content-Type: application/json
    • Return message contains the full representation of the resource
    • Client can control representation of the resource via attribute selector directive fields=field1,field2, ... In this case, requested fields will be presented. If fields=none, then response contains only id, href and name (also optionally @type, @schemaLocation as per API implementation).
    • For optional and not specified attributes of the resource model, entity is populated by default values as per entity schema specification.

    The following table depicts the critical response codes for this scenario. For details of response codes, please refer to Methods, Response Codes and Error Representation.

    Response Codes-Success Response Codes-Error
    • 201 Created (if SYNC)
    • 202 Accepted (if ASYNC)
    • 400 Bad Request
    • 404 Not Found
    • 405 Method Not Allowed
    • 409 Conflict
    • 5XX Server Errors

    1.2. Creating multiple resources

    http-PATCH method is used for creation of multiple resources. The following design guidelines apply for this pattern:

    • Content-Type: application/json-patch+json
    • PATCH operation is relative to the container collection for the entity types to be created, where the following directives detail the type of update:
      • “op” is the action to take. For creating multiple resources at a given path, always "add".
      • “path” is a pointer/reference to the field to change.
      • “value” is the value object to add.
    • Return message contains the full representation of the resource
    • Client can control representation of the resource via attribute selector directive fields=field1,field2, ... In this case, requested fields will be presented. If fields=none, then response contains only id, href and name (also optionally @type, @schemaLocation as per API implementation).
    • For optional and not specified attributes of the resource model, entity is populated by default values as per entity schema specification.
    • The HTTP PATCH method is atomic, as per [RFC5789] i.e., all operations MUST be successful otherwise the application of the full PATCH is unsuccessful.

    The following table depicts the critical response codes for this scenario. For details of response codes, please refer to Methods, Response Codes and Error Representation.

    Response Codes-Success Response Codes-Error
    • 200 OK (if SYNC)
    • 202 Accepted (if ASYNC)
    • 204 No Content
    • 400 Bad Request
    • 404 Not Found
    • 405 Method Not Allowed
    • 409 Conflict
    • 5XX Server Errors

    2. Query Resources Patterns

    The fundamental concept in MonE RESTful API is the resource with media type "application/json". The following diagram depicts the two possible types of resources in a response to a query: whether a single object, or a collection/array of objects.

    Figure-1: Query Response: Resource Models

    This section details the query method patterns of resources in the following sections:

    • Querying a single resource
    • Querying multiple/collection resources
    • Querying multiple/collection resources with JSON Path Extension

    2.1. Querying a single resource

    http-GET method is used for querying of a single resource. The following design guidelines apply for this pattern:

    • GET {apiRoot}/{resourceName}/{resourceID} MUST be used to retrieve the representation of a Managed Resource named resourceID.
    • Return message contains the full representation of the resource
    • Client can control representation of the resource via attribute selector directive fields=field1,field2, ... In this case, requested fields will be presented. If fields=none, then response contains only id, href and name (also optionally @type, @schemaLocation as per API implementation).
    • If resource identified by URI cannot be identified 404 Not Found is returned.

    The following table depicts the critical response codes for this scenario. For details of response codes, please refer to Methods, Response Codes and Error Representation.

    Response Codes-Success Response Codes-Error
    • 200 OK
    • 400 Bad Request
    • 404 Not Found
    • 405 Method Not Allowed
    • 5XX Server Errors

    2.2. Querying multiple resources

    http-GET method is used for querying of multiple resources. The following design guidelines apply for this pattern:

    • GET {apiRoot}/{resourceName} MUST be used to retrieve the representation of a Collection Resource
    • Return message contains the array of one or more Managed Resource entities that constitute the Collection Resource
    • For query-based pagination, the HTTP header “X-Total-Count” in a response is returned, with the total number of matching resources so that the client can calculate the next page.
      • offset and limit parameters are used to control pagination by the client.
    • If no paging query parameters are specified in the URI then all matching resources are returned to the consumer, considering the default default paging values for offset and limit.
      • If the response can fit a single page, response code returned is 200
      • If the response cannot fit a single page, then the response code returned is 206.
    • In the case of pagination, the responses contain Link HTTP header that allows a way for the server to communicate the client a menu for possible future requests from which the client is free to choose, such as "first", "next", "prev", and "last" page related URLs.
    • Each entity in return message contains the full representation of the resource
    • Client can control representation of the entity resources via attribute selector directive fields=field1,field2, ... In this case, requested fields will be presented. If fields=none, then response contains only id, href and name (also optionally @type, @schemaLocation as per API implementation).
    • If resource identified by URI cannot be identified 404 Not Found is returned.

    The following table depicts the critical response codes for this scenario. For details of response codes, please refer to Methods, Response Codes and Error Representation.

    Response Codes-Success Response Codes-Error
    • 200 OK
    • 206 Partial Content (returned in case of pagination)
    • 400 Bad Request
    • 404 Not Found
    • 405 Method Not Allowed
    • 5XX Server Errors

    2.2.3. Query Multiple Resources: Sorting

    sort=[direction]field, [direction]field,...

    where:

    • direction : is optional. Default is "+" indicating ascending. Can be: “-“ | “+”
    • field: The field to sort on.

    The following table depicts sorting control related query parameters usage:

    ?sort=name Sort the result set on the name in ascending order.

    ?sort=-name Sort the result set on the name in descending order.
    ?sort=name,state Sort the result set on the name in ascending, then by state in ascending order.
    ?sort=-name,state Sort the result set on the name in descending, then by state in ascending order.

    2.2.4. Query Multiple Resources: Conditional Collection Filtering

    This section describes the TMFORUM open API basic design patterns for conditional attribute filtering with less flexibility. Please note that unless otherwise explicitly specified in section MonE Public APIs Specification, all MonE Public API supports JSONPath based conditional attribute filtering as detailed in next section.

    It is possible to have conditional collection items filtering, while issuing query multiple resources. The basic expression is one or a sequence of ANDED attribute assertions to formulate a filtering expression as the following usage patterns:

    • SINGLE CONDITION: GET {apiRoot}/{resourceName}?{attributeName1}={attributeValue1}
    • MULTIPLE CONDITIONs: GET {apiRoot}/{resourceName}?{attributeName1}={attributeValue1}&{attributeName2}={attributeValue2}...

    The following table depicts the operators that may be used:

    Operator URL Encoded
    Form    		 Suffix
    Form    		 Description
    = %3D%3D .eq equal to ( = )
    Return results where the search criteria field is equal to
    > %3E .gt greater than ( > )
    Return results where the search criteria field is strictly greater than
    ≥ %3E%3D .gte greater than or equal to ( >= )
    Return results where the search criteria field is equal to or greater than
    < %3C .lt less than ( > )
    Return results where the search criteria field is strictly less than
    ≤ %3C%3D .lte less than or equal to ( >= )
    Return results where the search criteria field is equal to or less than
    *= %3D~ .regex Regex expression
    Note: all regex special characters must be encoded


    The following table depicts conditional attribute filtering usage examples. Note that some examples provide both types of usage via Suffix Form or URL encoded form of the operator presented in the table above.

    Query Description

    2.3. Querying Multiple Resources with JSON Path Extension

    This section describes MonE API JSON Path support for querying multiple resources adhering to TMFORUM open API JSON Path based design patterns for conditional attribute filtering as detailed in (Ref. [1], Part-6). Please note that unless otherwise explicitly specified in section MonE API Catalog & Specs, all MonE API supports JSONPath based conditional attribute filtering as detailed in this section.

    Please note that PAGINATION logic and directives are as described in Query Multiple Resources: With Pagination.

    http-GET method is used for querying of multiple resources. The following design guidelines apply for this pattern:

    • GET {apiRoot}/{resourceName} MUST be used to retrieve the representation of a Collection Resource
    • Return message contains the array of one or more Managed Resource entities that constitute the Collection Resource
    • For query-based pagination, the HTTP header “X-Total-Count” in a response is returned, with the total number of matching resources so that the client can calculate the next page.
      • offset and limit parameters are used to control pagination by the client.
    • If no paging query parameters are specified in the URI then all matching resources are returned to the consumer, considering the default default paging values for offset and limit.
      • If the response can fit a single page, response code returned is 200
      • If the response cannot fit a single page, then the response code returned is 206.
    • Each entity in return message contains the full representation of the resource
    • Client can control representation of the entity resources via attribute selector directive fields={JSONPath*}. In this case, requested fields as per provided JSONPath will be presented. As a shortcut, if fields=none, then response contains only id, href and name (also optionally @type, @schemaLocation as per API implementation).
    • For conditional collection filtering, filter={JSONPathExpression} query directive is used as described in this section, instead of the pattern as described in previous section Query Multiple Resources: Conditional Collection Filtering.
    • If resource identified by URI cannot be identified 404 Not Found is returned.

    2.3.1. JSONPath Overview

    The equivalent of XPath for JSON documents is called JSON Path. It is a query-oriented language that allows querying and extraction of subsection(s) of a JSON document.

    2.3.1.1. JSONPath Expression & Operators

    The following table depicts the JSON Path expression and operators:

    Expression & Operator Description
    $ Root object/element to query. This is how all the path expressions are started.
    @ The current node being processed - it is used in the input expressions for filter predicates.
    .<attribute> Child selector - selects the specified property in a parent object using the "dot" (.) notation.
    ['<attribute>(,'<attribute>')] Child or children selector - selects the specified property or properties in a parent object using the "bracket" ([) notation.
    Note: This expression SHOULD be used if the attribute contains special characters such as spaces, or begins with a character other than A..Za..z_.
    .. Recursive descent.
    It will search for the specified attribute name recursively and it will return an array of all values with the attribute name.
    It will always return a list, even if only one property is found.
    * Wildcard selector.
    It will select all elements (object or array) regardless of their names or indexes.
    For example, $.relatedEntity[0][*] means all attributes of the first object from the relatedEntity array, and $.relatedEntity[*] means all items of the relatedEntity array.
    [n] Selects the nth element from an array. Indexes are 0-based.
    [indexNr,
    indexNr, ...]    		 Selects array elements with the specified indexes. Returns a list.
    [start:end]
    [start:]    		 Array slice operator.
    It will select array elements from the start index and up to, but not including, end index.
    If the end is omitted, selects all elements from start until the end of the array. Returns a list.
    [:n] Selects the first n elements of the array. Returns a list.
    [-n:] Selects the last n elements of the array. Returns a list.
    [,] Union operator. JSONPath allows alternate names or array indices as a set.
    [?(expression)] Filter expression.
    Selects all elements in an object or array that match the specified filter.
    [(expression)] Script expression.
    Script expressions can be used instead of explicit property names or indexes. An example is [(@.length-1)] which selects the last item in an array. Here, length refers to the length of the current array rather than a JSON field named length.

    2.3.1.2. JSONPath Filter Expression Predicates

    JSONPath predicate filters are Boolean expressions (true or false) that restrict returned lists of nodes - usually applied to arrays. The filter syntax is: 

    [?(expression)]

    As detailed in previous section regarding JSONPath operators, the filter syntax is generally used in conjunction with the "@" operator (@ pointier to current item being processed) to create the predicate.

    If predicate filter does NOT validate, an empty response is returned.

    Example

    JSON Path Expression

    RESULT

  }

    RESULT

    "status" : "Resolved",
    "changeReason" : "Issue has been resolved", 
    "changeDate" : "2018-05-02T00:00"
  }
]

    The following table depicts the JSON Path filter expression related operators:

    Operator Description
    == Equal to
    1 and '1' are considered equal. String values should be enclosed in single quotes or double-quotes.
    != Not equal to.
    > Greater than
    >= Greater than or Equal to
    < Less than
    <= Less than or Equal to
    =~ Match a JavaScript regular expression.
    For example, $.statusChange[?(@.status=~/Resol.*?/i)] matches items whose status starts with Resol.
    ! Negate filter condition.
    Use to negate a filter: $.attachment[?(!@.size)] matches items that do not have the size property.
    && Logical AND
    Used to combine multiple filter expressions: $.attachment[?(@.size==300 && @.sizeUnit=='KB')]
    || Logical OR
    Used to combine multiple filter expressions: $.attachment[?(@.size==300 || @.size==500)]

    2.3.3. Sorting with JSONPath

    To support sorting the sort directive is used as HTTP query parameter. The definition is as the following:

    sort=[direction]sort-field, [direction]sort-field,...

    where:

    • direction : is optional. Default is "+" indicating ascending. Can be: “-“ | “+”
    • sort-field: The field to sort on, specified as a JSONPath expression

    Example

    GET /api/troubleTicket?sort=channel.name
Access: application/json

    or

    GET /api/troubleTicket?sort=attachment[*].name
Access: application/json

    or

    GET /api/troubleTicket?filter=attachment[?(@.size==300)]&fields==['id','href','name']& sort=attachment[*].name
Access: application/json

    2.4. Query Depth and Expand Directives

    As for any REST based API, MonE API query responses, return single or a collection of resources in which each resource contain referenced entities. A common use case is to query a resource and “dereference” or “embed” the hyperlinked resource representations. Control of expansion in the hierarchy of referenced objects is controlled via the following two directives:

    • depth: The value of this directive controls the number of depth level subject to expansion in the hierarchy of references.
      • If depth=0, then no expansion, meaning all referenced entities and response is just RootEntity objects which contains only @type, @schemaLocation, id and href.
      • If depth = 1, then the first level of related target entities is expanded. References are replaced with the actual loaded entities with the corresponding level of details.
      • If depth = N it expands reference objects of related entities recursively. The last level contains only the references.
    • expand: It is optional and must be used together with depth. The value of this directive is used to narrow down (filter) expanding of the referenced objects.
      • If expand is not specified, then by default all the paths should be expanded as per specified depth value.
      • If the expand parameter is specified, only those paths should be expanded according to depth parameter.

    4. Delete Resources Patterns

    http-DELETE method is used for deletion/removal of a resource. The following design guidelines apply for this pattern:

    • Content-Type: application/json (for response). A Delete operation does NOT accept a request payload.

    The following table depicts the critical response codes for this scenario. For details of response codes, please refer to Methods, Response Codes and Error Representation.

    Response Codes-Success Response Codes-Error
    • 200 OK (if SYNC and if the action has been enacted and the response message includes a representation describing the status)
    • 202 Accepted (if ASYNC )
    • 204 No Content (if SYNC and if the action has been enacted and no further information is to be supplied)
    • 400 Bad Request
    • 404 Not Found
    • 405 Method Not Allowed
    • 409 Conflict
    • 5XX Server Errors
