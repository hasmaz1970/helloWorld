    MonE Private APIs Specification

    This section provides the Catalog of MonE Private API(s) specification as grouped by Reference Point(s). Reference Point style of classification and grouping of APIs allows better clarity of the bounded context from a documentation perspective when for a particular goal, multiple related interactions exist between an external client and MonE.

    Definition

    Reference Point(s): denote a systematic grouping/classification of related API(s) between any two components (as also used by 3GPP Technical Documents). Thus, a Reference Point, such as Mchg denotes/describes one or more API(s) for a bounded context between the Messaging Streaming Platform and MonE Charging components, allowing the required information exchange between charging components to realize calculating the rated amounts and discounts of events.

    Private API(s): denote a programmatic interface which defines interactions between MonE Components for a particular purpose. MonE Private APIs can be realized using different technologies and/or transport mechanisms such as: Message Bus APIs, DB APIs, In-Memory Cache Access APIs or http based information exchange between components.

    For a further detailed description of the modules and API Reference Point(s) within the overall Architecture, please refer to MonE L1 Architecture.

    The following table provides a catalog of MonE Public APIs specification.

    Important

    Please note for the diagram above and "API Style/Protocol" specified in the following table for Reference Points related to: Event/Message Streaming Platform and In-Memory Cache Platforms. Even though the Private APIs for these Reference Points may NOT be fully utilizing http-REST API Style/Protocol, for the sake of convenience, API documentation is still provided on Open API Spec 3 and Swagger UI.

    As an example; Mchg reference point uses Apache Kafka Messaging Streaming API with binary data streaming over TCP.

    For these (non http-REST) APIs, the interpretation of Swagger UI based API specification shall be as the following:

    • GET implies read from and POST implies write to the relevant platform.
    • Information Data Models specified in the Swagger UI are exact and show the actual information being exchanged.
    • Response codes are representational and follow http semantics (2xx success, 4xx/5xx error) to provide success and error cases for the reader. For actual success/error codes returned from the relevant platform, please refer to API Specification of that platform, such as: Apache Kafka for Event/Message Streaming Platform and Apache Ignite for In-Memory Cache Platform.
